# M4.4B Runtime Cleanup Audit

## Runtime Paths Audited

- `src/lib/cmo/app-chat-store.ts` owns product request normalization, app/workspace/user/session identity, source cache/context packaging, Hermes CMO invocation, turn persistence, and fallback selection.
- `src/lib/cmo/hermes-cmo-chat-mapper.ts` owns the product request envelope to Hermes CMO and maps validated Hermes responses back to the app-chat response shape.
- `src/lib/cmo/hermes-cmo-runtime.ts` owns Hermes response validation, safe classification allowlists, event/delegation validation, and forbidden mutation counter rejection.
- `src/lib/cmo/runtime.ts` owns legacy/local CMO runtime fallback, including local Source Review rendering.
- `src/lib/cmo/source-acquisition/*` owns chat-time source extraction/cache/context artifacts only. Chat source artifacts remain session-local and are not official 13 Sources writes.
- Legacy Surf/Echo direct paths remain in `app-chat-store.ts`, but only run when Hermes CMO is not used or after Hermes CMO fails/invalidates.

## Local Answer Synthesis Risks

- `FallbackRuntime.sourceReviewFallbackAnswer` can render `Source Review`, `What I Read`, and `CMO Read`. This is allowed only outside the Hermes-live success path.
- `runtime.runTurn` can produce local source answers from `sourceReviewContext` and `sourceAnswerContext`. In Hermes-live workspaces this is now marked as `fallback_after_hermes_failure`.
- Direct legacy Surf/Echo bridges can still produce final answers. In Hermes-live workspaces these are fallback-only because the Hermes branch runs first.
- Local session commands for reviewing decision-layer candidates remain product-shell controls, not source-gathering answers.

## Runtime Boundary

When Hermes CMO is requested and returns a valid response, CMO Engine renders Hermes output directly and marks the turn with:

- `productRenderSource: "hermes_cmo"`
- `hermesRequestSent: true`
- Hermes metadata `productRenderSource: "hermes_cmo"`

When Hermes CMO is requested but unavailable or invalid, CMO Engine may use the existing fallback path and marks the turn with:

- `productRenderSource: "fallback_after_hermes_failure"`
- `productFallbackReason`
- `hermesCmoStatus: "failed_then_existing_fallback"` or `guardrail_violation_then_existing_fallback`

When Hermes CMO is disabled or force-fallback is requested, legacy/local runtime and bridge paths remain available and are labeled as `legacy_cmo_engine`, `local_runtime_fallback`, or `direct_bridge`.

## Tool-First Source Reading

CMO Engine sends Hermes CMO the raw user message plus read-only tool policy for web, browser, file, read-only terminal, code execution, vision, skills, session search, clarify, todo, memory-read, and delegation. Messaging, cronjob, and kanban remain disabled. Durable writes still require explicit confirmation and approved product/Vault Agent flows.

## Source Acquisition Role

- Chat path: cache/context provider for Hermes CMO and fallback, with session-local artifacts only.
- Official durable ingestion: Inputs & Priorities / Sources UI future flow.
- Guardrails remain: no chat auto-save to 13 Sources, no 12 Knowledge promotion, no GBrain mutation.

## Future Cleanup Targets

- Split legacy Surf/Echo direct bridge code behind a non-Hermes gateway module so it cannot drift into the Hermes-live path.
- Replace local Source Review fallback copy with an explicitly labeled fallback UI/metadata component.
- Add an end-to-end replay fixture that stores a real Hermes request, replay result, product render source, and fallback reason for A/B/C/D diagnosis.
