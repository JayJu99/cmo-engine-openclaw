# M4.4A - CMO Source-Gathering Workspace Contract

Scope: CMO Engine side contract, harness, and boundary audit for the source-gathering pivot.

## Architecture Invariant

When Hermes CMO is enabled and live, CMO Engine is the product shell, smart gateway, session owner, and permission boundary. Hermes CMO is the source-gathering reasoning/tool agent. Vault Agent remains the safe durable memory and Vault boundary.

CMO Engine may:

- fetch and cache source artifacts for the current session;
- pass runtime context, source artifacts, Vault context packs, and tool policy to Hermes CMO;
- persist chat/session logs under the existing session pipeline;
- run fallback only when Hermes CMO is disabled, unavailable, or returns an invalid/unsafe response.

CMO Engine must not:

- synthesize final source-review/source-answer output while Hermes CMO is live;
- auto-save chat URL/file material into `13 Sources`;
- auto-promote material into `12 Knowledge`;
- mutate GBrain from chat source acquisition;
- bypass Hermes response validation or M1 mutation/delegation guardrails.

## Current Boundary Audit

- Hermes request envelope: `src/lib/cmo/hermes-cmo-chat-mapper.ts` builds `hermes.cmo.request.v1` with workspace, app, user, session, turn, runtime context, context pack, active source artifacts, Vault context pack artifact, and read-only source answer/review context.
- App/workspace/user/session identity: request identity is derived in `createAppChatSession` and mapped as `workspace`, `user`, `session_id`, and `turn_id`.
- Runtime context: `buildRuntimeContext` is attached to `contextPackage` and passed through `runtime_context`.
- Tool policy and allowed agents: `constraints` and `tool_policy` both express proposals-only, no local execution, no Vault/Supabase/session/raw-capture/OpenClaw writes, and allowed Echo/Surf boundaries.
- Source/session artifacts: session-local sources and source answer context are passed as read-only `artifacts_in`; chat cache is session-only and not an official project source.
- Hermes response validator: `src/lib/cmo/hermes-cmo-runtime.ts` rejects unknown classifications, unknown activity events, unsafe delegations, and mutation flags.
- Hermes mapper/render: `mapHermesCmoResponseToChatResult` maps Hermes answer bodies into app-chat shape. Native/source-answer/source-transform/save-intent responses render Hermes body directly.
- Fallback runtime: `FallbackRuntime` can render local Source Review, but only after the Hermes-live path is not used.
- Turn logging: all app chat turns still write session JSON and run existing session indexing / Vault Agent dry-run handoff.
- Source ingestion / 13 Sources: official source ingestion is separate under the Vault source ingestion route and future Inputs & Priorities/Sources UI. Chat source acquisition remains session-local cache/context.

## Risky Or Legacy Brain-Like Paths

- `FallbackRuntime.sourceReviewFallbackAnswer` renders local Source Review. This remains acceptable only for fallback/disabled Hermes paths.
- `buildSourceAnswerContext` performs local snippet selection. It should remain a context-provider/cache helper, not final answer synthesis.
- `routeIntentForMessage`, direct Surf/Echo bridge, mixed CMO/Echo, and CMO Surf evidence orchestration remain in the non-Hermes legacy path. They should not compete when Hermes chat is enabled and valid.
- Local decision-layer review commands still mutate review status. They are explicit product UI/session commands, not source-answer synthesis, but should stay outside Hermes source-review turns.

## Product Request Contract

CMO Engine sends Hermes:

- workspace/app/user/session/turn identity;
- `runtime_context`;
- `tool_policy` and `constraints` for delegation/write boundaries;
- active session-local source cache and source answer/review context;
- Vault context pack artifact when available;
- `session_context_pack: null` placeholder for later contract expansion;
- durable write constraints: no chat auto-save to `13 Sources`, no auto-promote to `12 Knowledge`, no GBrain mutation, Vault writes require explicit save flow;
- allowed delegation boundaries: proposals only by default, Echo/Surf allowed as bounded proposals, Vault Agent execution disabled from chat.

## Response Handling Contract

Accepted safe Hermes classifications:

- `native_conversation`
- `source_answer`
- `structured_review`
- `strategy_only`
- `external_research`
- `source_translate`
- `source_transform`
- `save_to_vault`

Unknown classifications, unknown events, forbidden delegations, and mutation/write flags are rejected by the runtime validator and cause the existing guarded fallback path.

## Future Cleanup Candidates

- Split `buildSourceAnswerContext` naming so it cannot be mistaken for final answer generation.
- Move direct Surf/Echo bridge paths behind the same product boundary policy or retire them for Hermes-live apps.
- Replace local Source Review fallback with a clearly labeled fallback-only component once Hermes coverage is stable.
- Add the future Inputs & Priorities / Sources UI explicit Save Source flow as the only official `13 Sources` ingestion entrypoint.
